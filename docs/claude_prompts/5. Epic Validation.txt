## What validation means

### Epic validation checks
Validate whether the epic:
- is correctly bounded
- maps cleanly to the baseline architecture
- does not hide scope from another epic
- has clear dependencies
- has strong acceptance criteria
- has explicit testing expectations
- is appropriate as an implementation planning unit

### Story validation checks
Validate whether each story:
- belongs to this epic and only this epic
- is small enough to implement safely
- is large enough to be meaningful
- has clear scope in / scope out
- is testable
- has strong regression/invariant coverage expectations
- is traceable to epic + PRD + RFC + ADR
- does not smuggle in task-level implementation decisions unnecessarily
- does not overlap excessively with sibling stories
- does not miss key edge cases

## Validation statuses to use
Use only these statuses for this pass:

- `draft` = generated but not yet approved
- `validated` = approved for downstream decomposition
- `needs_revision` = close, but not yet good enough
- `superseded` = replaced by another item

You may recommend a later `ready` status, but do not assign `ready` in this step.
