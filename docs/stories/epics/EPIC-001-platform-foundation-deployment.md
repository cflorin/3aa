# EPIC-001 — Platform Foundation & Deployment

## Purpose
Establish cloud infrastructure, database schema, deployment pipeline, and development environment to support all V1 features. This epic delivers the foundational platform on which the modular monolith will run.

## Outcome
A fully operational GCP environment with:
- GitHub repository (version control, releases/tags)
- Cloud Run service (auto-scaling, HTTPS)
- Cloud SQL Postgres database (with all V1 tables)
- Cloud Scheduler configured for nightly batch
- Cloud Build CI/CD pipeline (GitHub integration)
- Secret Manager for credentials
- Database migrations framework operational
- Framework configuration seeded (anchored thresholds, TSR hurdles)
- Development environment replicable

## Scope In
- GitHub repository setup (main branch, .gitignore, version tags/releases)
- Git workflow configuration (branch protection, commit conventions)
- Semantic versioning strategy (v1.0.0, v1.1.0 tags for releases)
- GCP project setup (Cloud Run, Cloud SQL, Cloud Scheduler, Secret Manager, Cloud Build, VPC Connector)
- Database schema creation (Prisma migrations for all RFC-002 tables)
- Framework configuration seed data (anchored thresholds table: 16 codes, TSR hurdles table: 7 buckets, framework_version)
- Deployment pipeline (GitHub push → Cloud Build trigger → Docker build → Cloud Run deploy)
- Database migration tooling (Prisma migrate)
- Development environment setup (local Postgres, env vars, Docker, Git)
- Infrastructure-as-code (Terraform or gcloud scripts, committed to Git)
- GCP IAM configuration (service accounts, least privilege)
- Network configuration (VPC Connector for Cloud Run → Cloud SQL private connection)
- Health check endpoint (`/api/health`)

## Scope Out
- Application business logic (classification, valuation, alerts)
- User-facing features or UI screens
- Third-party observability platforms (Sentry - deferred)
- Multi-region deployment (V1 is single-region: us-central1)
- Load balancing / CDN (Cloud Run handles this)

## Test Strategy Expectations

**Unit tests:**
- Prisma schema validation (schema compiles without errors, types generated)
- Migration scripts idempotent (can run multiple times safely without errors)
- Seed data validation (anchored thresholds: 16 rows, thresholds descending order for each code)
- Seed data validation (TSR hurdles: 7 rows, buckets 1-7 present)
- Health check endpoint logic (returns correct JSON structure)

**Integration tests:**
- Database connection test (Next.js app can connect to Cloud SQL, query users table)
- Migrations apply successfully (create test database, apply migrations, verify all tables exist)
- Seed data queryable (fetch anchored thresholds WHERE code='4AA', verify max=22, comfortable=20, very_good=18, steal=16)
- Seed data queryable (fetch TSR hurdle WHERE bucket=4, verify base_hurdle_default=12.5)
- Secret Manager read test (app can read DATABASE_URL secret)
- Foreign key constraints enforced (attempt to insert classification_state with non-existent ticker → fails)
- Unique constraints enforced (attempt to insert duplicate anchored threshold code → fails)

**Contract/schema tests:**
- RFC-002 schema compliance (Prisma schema matches RFC-002 table definitions exactly)
- Prisma schema → database schema parity (generated SQL matches expected schema)
- Framework config integrity (anchored thresholds table: 16 rows, codes match framework doc)
- Framework config integrity (TSR hurdles table: 7 rows, adjustment values match framework doc)
- Foreign key cascade behavior (delete stock → classification_state, valuation_state cascade deleted)

**BDD acceptance tests:**
- "Given a fresh database, when migrations run, then all 17 tables exist"
- "Given framework seed data, when querying anchored_thresholds, then 16 codes present"
- "Given framework seed data, when querying tsr_hurdles, then 7 buckets present"
- "Given Cloud Run deployment, when health check called, then 200 OK with JSON payload returned"
- "Given valid DATABASE_URL, when app starts, then Prisma connects successfully"

**E2E tests:**
- Full deployment flow (git push to main → Cloud Build triggers → Docker builds → Cloud Run deploys → health check passes)
- Cloud Scheduler trigger test (manually trigger price-sync job → placeholder endpoint returns 200 OK)

## Regression / Invariant Risks

**Schema drift:**
- Risk: Prisma schema diverges from RFC-002 canonical definition
- Protection: Contract tests validate schema compliance, CI fails if mismatch

**Seed data corruption:**
- Risk: Framework config manually edited, violates thresholds descending order
- Protection: Seed data validation tests, CHECK constraints on thresholds table

**Migration rollback failures:**
- Risk: Forward migration succeeds but rollback breaks database
- Protection: Test rollback in CI, maintain migration history

**Secret rotation breaks app:**
- Risk: Database credentials changed in Secret Manager, app connection fails
- Protection: Secret Manager versioning, graceful connection error handling

**Cloud Run cold start timeout:**
- Risk: First request after idle exceeds timeout, health check fails
- Protection: Min instances = 0 acceptable for V1, monitor cold start latency

**Invariants to protect:**
- Database schema matches RFC-002 exactly (all tables, columns, constraints, indexes)
- Framework config immutable for V1 (anchored thresholds, TSR hurdles frozen)
- Database foreign key cascades work correctly (deleting stock cascades to classification_state, valuation_state, alerts)
- Migrations are reversible (every migration has up + down)
- Cloud Run can always connect to Cloud SQL (VPC Connector never deleted)
- Anchored thresholds always in descending order (max > comfortable > very_good > steal)
- TSR hurdles complete for buckets 1-7 (no missing buckets)

## Key Risks / Edge Cases

**Infrastructure risks:**
- GCP quota limits exceeded (Cloud Run instances, Cloud SQL connections)
- Cloud SQL instance undersized (db-f1-micro insufficient for V1 load)
- VPC Connector misconfiguration (Cloud Run cannot reach Cloud SQL)
- Cloud Scheduler OIDC auth fails (service account permissions incorrect)
- Cloud Build timeout (Docker build exceeds 10 min limit)

**Database risks:**
- Prisma migration fails mid-apply (database in inconsistent state, manual intervention required)
- Seed data violates constraints (thresholds not descending, INSERT fails)
- Index creation timeout (CREATE INDEX on large table exceeds statement timeout)
- Connection pool exhaustion (Prisma client pool size too small, queries fail)

**Deployment risks:**
- Cloud Run deployment failure (container crashes on startup, missing env var)
- Environment variable misconfiguration (DATABASE_URL incorrect format, connection fails)
- Docker image too large (exceeds Cloud Run size limit)

**Edge cases:**
- Cold start latency (first request after idle takes >5s, acceptable for V1)
- Concurrent migration execution (two developers run migrations simultaneously, conflict)
- Database connection pool exhaustion (100 concurrent requests, pool size 10)
- Framework config update (anchored thresholds changed, requires new migration + framework_version bump)
